# 前端 Auth 与游客门禁 Harness 规范

## 目标

本文档用于定义 KOC 项目前端在多账号部署阶段的登录态、未登录态和游客体验边界。

覆盖对象：

- [frontend/context/AuthContext.tsx](../../../../frontend/context/AuthContext.tsx)
- [frontend/context/AppStateContext.tsx](../../../../frontend/context/AppStateContext.tsx)
- [frontend/components/RequireAuth.tsx](../../../../frontend/components/RequireAuth.tsx)
- [frontend/components/LoginButton.tsx](../../../../frontend/components/LoginButton.tsx)
- [frontend/components/Sidebar.tsx](../../../../frontend/components/Sidebar.tsx)
- [frontend/app/login/page.tsx](../../../../frontend/app/login/page.tsx)
- [frontend/app/profile/page.tsx](../../../../frontend/app/profile/page.tsx)
- [frontend/app/trending/page.tsx](../../../../frontend/app/trending/page.tsx)
- [frontend/app/content/page.tsx](../../../../frontend/app/content/page.tsx)
- [frontend/app/page.tsx](../../../../frontend/app/page.tsx)

本文档约束的是：

- 登录入口与退出入口的位置
- 未登录游客能做什么、不能做什么
- 热门追踪和内容撰写的登录门禁
- 人设打造的游客试用边界
- 登录态变化时前端状态如何清理和恢复
- 聊天输入、消息数量与后端上下文传递的前端限制

它不是一次性的 UI 备注，而是 `docs/harness/` 体系中的前端 auth harness 实现规范。

## 适用范围

当前正式聊天页面体系包括：

- 人设打造 `profile`
- 热门追踪 `trending`
- 内容撰写 `content`
- 灵光一闪 `/?view=dialog`

说明：

- 人设、热门、内容是当前三条正式业务主线。
- 灵光一闪当前是正式聊天页面，但不承载保存链路，也不承载结构化业务结果。
- `/data`、`/operation` 当前不属于本文档重点覆盖范围。
- 本文档同时约束登录页、认证弹窗和左侧栏账号入口。

## 总体原则

多账号部署后，前端必须把页面状态分成两类：

1. 当前登录用户的业务数据
2. 当前浏览器的临时会话现场

统一原则：

- 登录用户的数据必须来自当前 session，不来自前端传入的 `userId`。
- 未登录游客只能试用人设首轮生成，不能保存任何业务结果。
- 热门追踪和内容撰写必须登录后才可使用。
- 登录、退出、切换账号后，前端不得继续展示上一个账号的业务数据。
- 前端可以做输入长度和可见消息数量限制，但这不能替代后端身份隔离。

## Auth 状态模型

前端统一使用三态登录模型：

```text
loading
authenticated
anonymous
```

语义如下：

| 状态 | 含义 | 页面行为 |
| --- | --- | --- |
| `loading` | 正在通过 `/api/auth/me` 确认 cookie session | 不提前展示受保护数据 |
| `authenticated` | 当前浏览器持有有效 `koc_session` | 允许访问完整业务模块 |
| `anonymous` | 未登录或 session 失效 | 只允许游客能力 |

规则：

- 页面首次加载时必须先走 `loading`。
- `loading` 期间不应把用户当作已登录，也不应把用户当作游客直接触发生成。
- `/api/auth/me` 返回 401 后进入 `anonymous`。
- 登录、注册成功后必须刷新当前用户状态。
- 退出登录后必须立即进入 `anonymous`，并清理当前前端用户数据。

## 登录入口规范

### 1. 右上角登录入口

以下正式业务主线页面的初始视图右上角应提供登录入口：

- 人设打造初始页
- 热门追踪初始页
- 内容撰写初始页

统一要求：

- 登录按钮文案使用 `登录`。
- 登录按钮走统一 [LoginButton.tsx](../../../../frontend/components/LoginButton.tsx)。
- 登录链接必须带 `next` 参数，登录成功后回到用户原本所在页面。
- 登录入口位置应和热门追踪、内容撰写保持同一视觉层级；人设打造初始页不能缺失。

推荐跳转：

```text
/login?next=/profile
/login?next=/trending
/login?next=/content
```

补充：

- 灵光一闪如果停留在首页说明态，当前可以继续复用已有首页登录入口，不强制要求复制三条业务主线初始页的同构布局。
- 但它作为正式聊天页面，仍必须接入统一 `AuthContext` 状态与登录后跳转语义。

### 2. 登录页

登录页应同时支持：

- 登录已有账号
- 注册新账号并提示用户继续登录

规则：

- 注册成功后应给出明确成功提示，并允许用户继续进入登录流程。
- 登录成功后优先跳回 `next` 指向的站内路径。
- `next` 只允许站内相对路径，不能允许 `//` 或外部 URL。
- 登录失败必须展示明确错误，不应静默停留。

## 退出入口规范

退出入口固定放在左侧栏底部的大圆按钮上。

交互规则：

1. 已登录时，大圆按钮表示账号菜单。
2. 点击大圆按钮后出现二级菜单。
3. 二级菜单中提供 `退出登录`。
4. 点击 `退出登录` 后调用 `/api/auth/logout`。
5. 无论接口是否短暂失败，前端都应清理本地 auth 状态并回到未登录态。

未登录时：

- 左侧栏底部大圆按钮保持临时对话入口语义。
- 不展示退出登录菜单。

## 游客能力边界

未登录游客只允许体验一次人设生成。

允许：

- 打开人设打造页面。
- 填写基础信息。
- 发起 1 次 `persona.analyze`。
- 在当前页面看到生成结果。

不允许：

- 保存人设。
- 继续人设追问。
- 使用热门追踪。
- 使用内容撰写。
- 保存趋势记录。
- 保存内容草稿。

当前推荐前端记录方式：

```text
localStorage: koc-agent-anonymous-persona-generated = "1"
```

规则：

- 该 key 只用于限制当前浏览器游客试用次数。
- 该 key 不代表真实用户身份。
- 用户登录后不应继续受游客次数限制影响。
- 游客生成结果可以在当前页面展示，但不应作为正式已保存人设进入全局用户数据。

游客再次点击人设生成时，提示文案应表达清楚：

```text
未登录状态只支持 1 次人设生成。登录后可以继续完善并保存人设。
```

游客点击保存人设时，提示文案应表达清楚：

```text
未登录状态不支持保存人设。请先登录后再保存。
```

## 受保护页面门禁

热门追踪和内容撰写必须登录后才可使用。

受保护页面：

- `/trending`
- `/content`

统一通过 [RequireAuth.tsx](../../../../frontend/components/RequireAuth.tsx) 承载登录门禁。

未登录访问热门追踪时，应展示：

```text
登录后开启热门追踪
热门追踪会结合你的账号方向和历史记录生成建议。请先登录，再继续使用这个模块。
```

未登录访问内容撰写时，应展示：

```text
登录后开启内容撰写
内容撰写会保存草稿并延续你的账号风格。请先登录，再继续使用这个模块。
```

规则：

- 未登录时不要渲染真实业务页面后再弹窗拦截。
- 未登录时不要先加载趋势历史、草稿历史或已保存人设。
- 未登录时不要调用热门追踪或内容撰写生成接口。
- 门禁页应提供登录按钮，并带回当前页面的 `next`。

## 登录后仍需人设门禁

热门追踪和内容撰写有两层门禁：

```text
先登录
再检查是否已有保存人设
```

顺序不能反过来。

规则：

1. 未登录用户访问 `/trending` 或 `/content`，先看到登录引导。
2. 已登录但没有保存人设，才看到“先完成人设打造”的引导。
3. 已登录且已有保存人设，才进入正式业务页面。

这样可以避免：

- 未登录用户看到“缺少人设”后误以为不需要登录。
- 前端为了检查人设而提前请求受保护数据。

## 登录态数据恢复与清理

前端全局业务状态由 [AppStateContext.tsx](../../../../frontend/context/AppStateContext.tsx) 负责。

登录后应拉取：

- `/api/persona/me`
- `/api/trends/history`
- `/api/content/history`

退出或 session 失效后应清理：

- 当前人设
- 趋势历史
- 内容草稿
- hydrating 状态

规则：

- 不再使用 `demo-user` 作为前端业务用户。
- 不再通过 URL 或请求体中的 `userId` 决定当前用户数据。
- 登录态切换后，不允许继续展示上一个账号的历史记录。
- 如果用户 A 退出后用户 B 登录，页面必须按用户 B 重新拉取数据。

## 聊天输入与消息数量限制

当前前端统一限制聊天输入最多 500 个字符。

适用输入：

- 人设追问输入
- 热门追踪输入
- 内容撰写主题输入
- 内容修改意见输入

规则：

- 计数应按用户可感知字符处理，而不是简单 UTF-16 长度。
- 超过 500 字时，前端应阻止发送并展示提示。
- 当前阶段后端不承担输入长度限制主责，但后续对外部署时仍建议补后端校验。

当前前端消息限制：

```text
当前会话最多允许 80 条正式聊天消息
真正发给后端的 conversationHistory 只带最近 12 条
```

对应实现语义：

- `CONVERSATION_MAX_MESSAGES = 80`
- `CONVERSATION_SEND_BLOCK_THRESHOLD = 79`
- `CONVERSATION_CONTEXT_MESSAGE_LIMIT = 12`

规则：

- 消息数按用户消息和 Agent 正常回复合计计算，不把运行中、停止、错误等状态反馈算作正式聊天消息。
- 当当前会话消息数 `>= 79` 时，前端必须禁止继续发送新用户消息，避免本轮用户消息和 Agent 回复超过 80 条上限。
- 到达上限后，输入框和发送按钮禁用，输入框 placeholder 显示：`当前对话已达到 80 条消息上限，请先保存或开启新对话后继续。`
- 到达上限时，页面顶部使用现有提醒样式显示同一句文案一次；开启新对话或清空当前会话后才能重新触发。
- 发给后端的 `conversationHistory` 不应超过最近 12 条。
- 超过 12 条的旧消息后续需要进入记忆压缩接口，但当前先只保留接口占位。

## 记忆压缩预留接口

当前暂不实现真实记忆压缩，但前端 harness 需要保留接口语义。

推荐结构：

```json
{
  "recentMessages": [],
  "compressedMemory": {
    "summary": "",
    "sourceMessageCount": 0
  }
}
```

当前阶段：

- `recentMessages` 发送最近 12 条。
- `compressedMemory` 可以为空摘要占位。
- 后端和 Agent 暂不依赖压缩摘要生成最终结果。

后续真正做记忆压缩时，应把旧消息压缩为：

- 用户长期偏好
- 已确认的人设约束
- 已生成过的选题或草稿方向
- 用户明确否定过的方向

不要把旧消息原文完整塞回后端。

## Prompt 调试入口边界

Prompt 调试页面不属于生产用户功能。

规则：

- 生产环境不展示 Prompt 调试入口。
- 生产环境不部署或不开放 debug route。
- `promptOverride`、debug key、调试 provider 等能力只允许用于内部生成调试。
- 真正生成阶段不依赖 Prompt 调试页面。

前端层面：

- Prompt 调试入口必须受特性开关控制。
- 默认生产配置下不显示。

## 对前端实现者的硬规则

1. 登录态是全局状态，不允许各页面各自判断一套。
2. 热门追踪和内容撰写必须先过 `RequireAuth`，再过人设门禁。
3. 人设打造允许游客首轮试用，但游客不能保存。
4. 登录按钮位置属于跨页面通用设置，修改时必须同时检查人设、热门、内容初始页。
5. 退出入口固定在左侧栏底部大圆按钮二级菜单，不在页面头部另做一套。
6. 退出或 session 失效后必须清理前端用户业务数据。
7. `conversationHistory` 发给后端时只带最近 12 条，超过部分走后续记忆压缩接口。
8. 输入 500 字限制属于聊天输入通用规则，不允许某页放开、某页限制。

## 当前文档落地结论

前端 auth harness 的核心结论是：

```text
游客只试用一次人设；完整业务必须登录；登录态由 session 确认；退出必须清数据；受保护页面先登录再检查人设。
```
