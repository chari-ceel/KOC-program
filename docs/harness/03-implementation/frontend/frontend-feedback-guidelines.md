# 前端成功反馈实现规范

## 目标

本文档不是测评 checklist，而是前端代码实现规范。

它约束的是：

- 保存成功
- 删除成功
- 后续同类“轻量成功反馈”
- 以及与之配套的页面级错误反馈边界

在代码里应该如何统一实现，避免每个页面各自写一套提示样式、位置、时长和动画。

当前这一规范属于 harness 文档体系的一部分，后续新增页面或修改交互时，应默认遵守，而不是把它当成一次性的 UI 调整。

## 适用范围

当前先覆盖：

- 人设打造 `persona`
- 热门追踪 `trend`
- 内容撰写 `content`

后续如果 `data`、`operation` 等页面也出现“按钮操作成功后给一个轻量提示”的需求，默认直接沿用本规范。

## 统一原则

### 1. 成功反馈必须统一

保存成功、删除成功这类提示不应由各页面自由发挥。

统一要求：

- 都使用同一套顶部提示组件
- 都放在页面顶部中间
- 都使用同一种轻量浮层视觉
- 都使用同一种短时出现后自动消失的节奏

### 2. 优先复用，不要页面内临时拼一段提示 UI

后续遇到同类需求时：

- 不要在某个页面里临时写一个 `p` 标签做提示
- 不要把成功提示嵌在聊天内容区中间
- 不要单独再做一套 modal / alert / snackbar

而应优先复用统一组件：

- [frontend/components/TopToast.tsx](../../../frontend/components/TopToast.tsx)

以及统一样式：

- [frontend/app/globals.css](../../../frontend/app/globals.css)

### 3. 提示组件属于通用设置

顶部提示组件不只是一个视觉零件，而是聊天页的通用设置。

这意味着：

- 修改提示位置、样式、停留时长、动作按钮时，必须同时检查 `profile`、`trending`、`content`
- 不允许一页使用顶部提示，另一页继续用局部内联提示或 `window.alert`
- 不允许错误提示和成功提示分裂成两套完全不同的页面协议

## 当前标准实现

### 组件

当前统一组件为：

- [frontend/components/TopToast.tsx](../../../frontend/components/TopToast.tsx)

组件职责：

- 接收一条短文案
- 固定渲染在页面顶部中间
- 不阻塞页面操作
- 只承载轻量成功提示，不承载复杂说明

### 动画与样式

当前统一动画类为：

- `koc-top-toast`

对应定义位置：

- [frontend/app/globals.css](../../../frontend/app/globals.css)

统一效果：

- 轻微侧滑进入
- 短暂停留
- 轻微侧滑退出
- 总时长约 2 秒

## 文案规范

成功提示文案应简洁直接，不写成长句。

优先使用：

- `保存成功`
- `删除成功`

规则：

- 不要写成过长的解释句
- 不要把提示写成段落说明
- 不要把非阻断成功提示做成二次确认弹窗

## 使用规范

### 1. 什么时候使用顶部成功提示

适用于：

- 点击保存按钮后操作成功
- 点击删除按钮后操作成功
- 其他“已完成且无需用户继续确认”的轻量成功动作

### 2. 什么时候不要用这套提示

不适用于：

- 删除前确认
- 需要用户立即处理的错误
- 需要展示长文本说明的状态
- Agent 生成中提示
- 停止输出提示
- 对话生成失败提示

这些场景应该继续使用：

- `window.confirm`
- 聊天上下文内的 `Agent 状态反馈`
- 页面级错误提示文案
- 更明确的错误态展示

而不是硬套“成功反馈”的成功态样式。

补充说明：

- 页面级错误提示可以继续复用 `TopToast` 作为容器
- 但必须使用错误态样式与错误态文案
- `状态反馈块` 不应把 `TopToast` 作为主展示容器
- Agent 请求失败后的“刷新 / 重试”主入口应放在 `状态反馈块` 组件内部
- `TopToast` 不应再承担“重试上一次 Agent 请求”这类职责
- 如果页面级错误提示仍带操作按钮，其语义也必须与主状态反馈一致，不能一处重试请求、一处整页 reload

## 页面接入规则

当前 `persona`、`trend`、`content` 页面都应遵循：

1. 操作成功后更新本地 / 远端状态
2. 触发统一成功提示
3. 约 2 秒后自动清空提示状态

后续新增页面时，同样按这个顺序处理。

不要出现：

- 某页保存成功后在聊天流里插一条提示
- 某页删除成功后没有反馈
- 某页成功提示停留时间明显更长
- 某页成功提示放在底部或局部区域

## 推荐接法

推荐模式：

1. 页面维护一个轻量提示状态，例如 `notice` / `saveStatus`
2. 页面提供一个统一方法，例如 `showNotice(message, autoHide)` 或 `showSaveStatus(message, autoHide)`
3. 成功后调用该方法，并传入 `true` 让它自动消失
4. 页面顶部挂统一 `TopToast`

当前可参考的实际接法：

- [frontend/app/profile/page.tsx](../../../frontend/app/profile/page.tsx)
- [frontend/app/trending/page.tsx](../../../frontend/app/trending/page.tsx)
- [frontend/app/content/page.tsx](../../../frontend/app/content/page.tsx)

## 后续扩展规则

如果未来要新增“复制成功”“发布成功”“归档成功”等提示：

- 先判断它是否属于“轻量成功反馈”
- 如果属于，默认复用 `TopToast`
- 保持顶部中间、短文案、约 2 秒自动消失

只有在业务性质明显不同的时候，才允许新增另一类反馈组件。

## 与 harness 的关系

这套规范同时对应两层：

1. 开发实现规范
2. harness 测评验收规范

开发阶段要按本文档实现。

测评阶段再按：

- [docs/harness/04-validation/harness-evaluation-plan.md](../../../../docs/harness/04-validation/harness-evaluation-plan.md)

检查是否真的做到一致。

结论：

- 本文档约束“以后要怎么写代码”
- `harness-evaluation-plan.md` 约束“写完之后怎么验收”
