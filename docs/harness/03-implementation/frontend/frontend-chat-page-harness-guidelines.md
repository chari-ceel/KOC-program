# 前端 Harness 聊天页面实现规范

## 目标

本文档用于把当前正式聊天页面的前端实现沉淀为 harness 文档规范。

覆盖对象：

- [frontend/app/profile/page.tsx](../../../../frontend/app/profile/page.tsx)
- [frontend/app/trending/page.tsx](../../../../frontend/app/trending/page.tsx)
- [frontend/app/content/page.tsx](../../../../frontend/app/content/page.tsx)
- [frontend/app/page.tsx](../../../../frontend/app/page.tsx)

本文档约束的是：

- 页面布局骨架
- 聊天区消息展示方式
- 结构化结果块如何落地
- 输入区、按钮区、加载态、错误态、成功反馈如何统一
- 哪些交互属于跨页面通用设置，修改时必须按正式聊天页面体系一起评估与落地
- 哪些允许按场景差异化，哪些不允许

它不是一次性的 UI 备注，而是 `docs/` 体系中的前端 harness 实现规范。

当前正式聊天页面体系包括：

- 人设打造 `profile`
- 热门追踪 `trending`
- 内容撰写 `content`
- 灵光一闪 `/?view=dialog`

说明：

- 本文档的“正式业务主线”仍主要指人设、热门、内容三页

- `/data`、`/operation` 当前不属于本文档覆盖范围
- 后续如果新增新的正式聊天页，默认先继承本文档，而不是重新发明一套页面结构

## 正式聊天页面统一结论

当前正式聊天页面虽然业务文案不同，但已经基本形成同一种页面骨架：

1. 页面有明确的 `viewMode`
2. 进入聊天前展示一个场景入口视图
3. 进入聊天后展示统一的三段式聊天布局
4. 用户消息在右侧气泡
5. assistant 消息在左侧内容卡片
6. 结构化结果时，在 assistant 卡片内渲染业务块
7. assistant 消息下方挂统一动作区
8. 底部固定输入区继续多轮对话
9. 会话现场保存在浏览器 session 存储中

结论：

- 人设、热门、内容三页已经形成稳定的正式业务主线页面协议
- 灵光一闪当前已纳入正式聊天页面体系，但仍处于“轻量接入通用能力”的阶段
- 正式聊天页面不应继续被写成彼此独立、互不相关的页面协议

## 通用设置定义

本项目中的“通用设置”指：

- 不是某个业务场景独有
- 会直接影响聊天页的基础交互语义
- 一旦修改，就应默认先检查 `profile`、`trending`、`content` 三页，并评估 `/?view=dialog` 是否也要同步

这类设置不应被当成单页优化，而应被视为 harness 级规则。

后续 AI 或开发者修改以下任一项时，必须同时检查三条正式业务主线页面是否一致：

- 输入框在生成中的可编辑性与提交规则
- 发送按钮在生成中的形态与语义
- 停止生成的交互方式
- 错误反馈的展示位置与组件
- 成功反馈的展示位置与组件
- 初始态发起请求失败时是否切换到聊天态
- 会话恢复时对失败态、生成态的处理
- 聊天页顶部信息条的起点、尺寸、间距与透明容器规则

如果改动属于纯通用聊天能力，而不是结构化结果、保存链路或业务入口态特例，还必须额外检查灵光一闪是否需要同步。

规则：

1. 只要变更落在上述范围内，就默认属于“通用设置”。
2. 修改时必须同时评估 `profile`、`trending`、`content` 三条正式业务主线页面。
3. 如果修改的是纯通用聊天能力，还必须同时评估 `/?view=dialog`。
4. 如果当前只改了其中一页，除非用户明确要求只改单页，否则视为未完成。
5. 文档、实现、验收口径都必须同步更新，而不是只改代码。

## 页面与共用元素映射

为了避免后续修改时再次混淆，当前正式聊天页面里“稳定的三条业务主线页面”和“灵光一闪轻量聊天页”的共用元素与差异边界明确如下。

覆盖页面：

- [frontend/app/profile/page.tsx](../../../../frontend/app/profile/page.tsx)
- [frontend/app/trending/page.tsx](../../../../frontend/app/trending/page.tsx)
- [frontend/app/content/page.tsx](../../../../frontend/app/content/page.tsx)
- [frontend/app/page.tsx](../../../../frontend/app/page.tsx)

### 1. 正式聊天页面共用的基础元素

- 文本渲染组件：`MarkdownText`
- 消息动作组件：`MessageActions`
- 顶部提示组件：`TopToast`
- 聊天页顶部信息条套件：头像圆徽标 + 标题 + 一行说明 + 右侧返回按钮
- 会话态页面骨架：`viewMode + 头部 + 消息滚动区 + 底部输入区`
- session 级恢复能力：当前页聊天现场恢复

说明：

- 上述能力当前应优先视为正式聊天页面体系的通用能力。
- 对人设、热门、内容三页要求更强一致性。
- 灵光一闪至少应接入其中的最小聊天闭环能力，而不是维持一套完全独立的实现。

### 2. 有前置门禁的共用元素

- `trending`
- `content`

这两页必须复用：

- [frontend/components/RequirePersona.tsx](../../../../frontend/components/RequirePersona.tsx)

`profile` 不使用这个门禁，因此这项不是三条正式业务主线全共用，而是“部分页面共用”。

### 3. 三条正式业务主线共用的交互设置

以下交互一旦修改，默认视为三条正式业务主线共享：

- 生成中输入框的可编辑性与提交规则
- 生成中发送按钮是否切换为停止按钮
- 是否允许中断当前生成
- 停止生成后的状态恢复方式
- 错误提示的位置、组件、动作按钮
- 成功提示的位置、组件、停留时长
- assistant 消息动作区的按钮种类、顺序、间距与样式
- 首次请求失败时是否进入聊天态
- 聊天中失败时是否保留当前上下文

当前统一规则补充如下：

- 生成中输入框保持可编辑，不应禁用
- 生成中允许用户继续输入新内容，但这段内容只保留在输入框中，不应直接发出
- 生成中发送按钮统一切换为圆圈停止按钮
- 只有两种情况允许把新输入真正发出：
  - 当前 Agent 已自然生成完成
  - 用户主动点击圆圈停止当前输出后，再重新发送
- 生成中按回车不应偷偷发送，也不应把“停止当前输出”绑定到回车上
- 三个正式聊天页顶部信息条应复用同一套视觉骨架：
  - 外层容器默认透明，不承担卡片背景
  - 左侧起点需与下方 assistant 输出内容左起点对齐
  - 左侧圆形徽标、标题字号、说明字号、图标间距保持统一
  - 左侧圆形徽标使用统一实现，不允许三页分别手写：
    - 尺寸固定为 `72px * 72px`
    - 中心文字“梨”字号固定为 `34px`
    - 背景色固定为 `#EFE5D9`，透明度 `80%`，即 `rgba(239,229,217,0.8)`
  - 右侧返回按钮保留在同一行右端
  - `profile`、`trending`、`content` 任一页修改这套顶部信息条时，其余两页必须同步评估并对齐
- assistant 消息动作区当前统一通过 `MessageActions` 落地：
  - 按钮必须使用同一套圆形图标按钮样式
  - 当同时存在刷新、复制、保存时，顺序固定为“刷新 -> 复制 -> 保存”
  - 三个按钮之间的水平间距保持一致
  - 按钮悬浮提示应由前端自绘 tooltip 统一实现，不依赖浏览器原生 `title`，避免 Windows / macOS 行为不一致
  - 这套顺序与间距属于正式聊天页面体系的通用设置，`profile`、`trending`、`content` 和 `/?view=dialog` 都应同步评估

### 4. 可按场景分化但仍需在 harness 里声明的元素

以下内容允许分化，但必须在实现时明确知道它们不是通用设置：

- 聊天前视图入口类型
- 结构化结果 renderer
- placeholder 文案
- assistant 的业务身份文案
- 保存目标和保存条件
- suggestion chips 是否展示

原则：

- 共用元素先统一
- 分化元素再按场景创新
- 不允许在未判断前，直接把通用元素当成单页实现

## 改动前判断流程

以后只要修改 `profile / trending / content` 任一聊天页，必须先走下面这套判断流程。

### Step 1. 先判断改动对象是什么

先回答：

- 这次改的是页面骨架吗
- 改的是消息区公共行为吗
- 改的是输入区公共行为吗
- 改的是提示组件或提示逻辑吗
- 改的是会话恢复逻辑吗
- 改的是某个场景自己的结构化展示吗

### Step 2. 判断是否属于通用设置

如果改动落在以下范围，直接判定为“通用设置”：

- `MarkdownText`
- `MessageActions`
- `TopToast`
- 聊天页输入区行为
- 生成中状态行为
- 停止生成行为
- 错误提示行为
- 成功提示行为
- 首次失败是否切换聊天态
- 会话恢复基础规则

一旦属于通用设置：

1. 先检查三页是否都受影响
2. 再同步修改三页
3. 最后更新 harness 文档说明

### Step 3. 只有不属于通用设置，才允许做场景创新

如果改动对象是以下内容，才允许只改单页：

- 该页专属的聊天前入口
- 该页专属的结构化 renderer
- 该页专属的业务字段展示
- 该页专属的保存数据结构
- 该页专属的建议项内容

这类改动允许创新，但前提是：

- 不破坏共用元素
- 不绕开通用设置
- 不偷改三条正式业务主线共享交互语义

### Step 4. 实施顺序要求

对聊天页做改动时，推荐执行顺序固定为：

1. 先判断是不是通用设置
2. 如果是，先看三页
3. 如果不是，再看当前页的场景差异
4. 实现后回看 harness 文档是否需要补充

## 对 AI / 开发者的硬规则

以后处理聊天页需求时，必须默认遵守：

1. 先查 harness 中“页面与共用元素映射”。
2. 先查这次改动是不是命中了通用设置。
3. 命中通用设置时，不允许只改单页后就结束。
4. 只有确认不属于通用设置，才允许做单页创新或其他差异化实现。

## 页面骨架规范

### 1. 必须有 `viewMode`

每个聊天页必须显式区分“聊天前视图”和“聊天中视图”。

当前对应关系：

- 人设页：`form | chat`
- 热门页：`history | chat`
- 内容页：`drafts | chat`

要求：

- 非聊天态负责承载该场景的入口信息
- 聊天态负责承载正式多轮对话
- 不要把列表页、表单页、聊天页混在同一滚动区域里

### 2. 聊天态必须采用三段式布局

聊天态统一采用：

```text
顶部场景头部
中部消息滚动区
底部输入区
```

具体约束：

- 顶部头部：显示场景身份、简短说明、返回按钮
- 中部滚动区：只承载消息流与消息级操作
- 底部输入区：只承载当前轮输入，不夹杂业务说明大段文案

当前三条正式业务主线页面都已遵循这一结构，后续不允许把重要交互散落到别的区域。

### 3. 页面宽度与主容器应保持同一视觉级别

当前实现已经收敛到近似同一宽度层级：

- 顶层容器使用 `flex h-full w-full flex-col overflow-hidden`
- 聊天主内容一般收敛在 `max-w-[940px]`
- 非聊天态主内容一般收敛在 `max-w-[900px] ~ max-w-[980px]`

规范要求：

- 新场景应延续这一中心收束布局
- 不要让某一页聊天区突然变成超宽表格式布局
- 不要让输入框宽度与消息区完全脱节

### 4. 浏览器缩放与可视区适配属于 harness 级布局约束

聊天页和聊天前视图都必须对浏览器缩放后的有效可视区负责。

这里的“缩放”包括但不限于：

- 浏览器缩放到 `80%`
- 浏览器缩放到 `125%`
- 浏览器缩放到 `150%`
- 缩放后导致有效视口宽高明显变小的场景

规范要求：

- 不能因为固定宽度、固定 padding、固定字号或固定间距，让核心模块被直接挤出当前页面可视区
- 不能出现“卡片下半截、主按钮或已展开结果区被裁掉，但用户又没有明确滚动入口”的状态
- 不能依赖 `body/html` 整页滚动去兜底暴露被裁切内容
- 应优先让当前场景内容区在缩放后自适应收缩，包括：
  - 合理降低横向 padding
  - 让标签与输入区在窄视口下改为纵向排布
  - 让标题、说明和按钮区维持可读但不过度撑高
- 如果内容在当前可视高度下确实放不下，必须只在当前场景内容区提供纵向滚动，而不是让外层页面直接溢出
- 局部滚动出现后，用户必须仍然能访问到完整表单、主要 CTA 和展开后的补充信息

当前落地要求：

- `profile` 的人设打造初始页必须满足这一规则，尤其是“查看已保存人设信息”展开后不能把底部内容裁掉
- `trending`、`content` 后续新增或修改聊天前视图时，也默认继承同一缩放适配约束

## 聊天前视图规范

聊天前视图允许按场景承载不同业务入口，但结构上要遵守统一模式：

- 顶部一段场景欢迎语
- 一个核心入口组件
- 一个和该场景强相关的历史 / 草稿 / 表单区

当前落点：

- 人设页：信息表单
- 热门页：历史追踪记录列表
- 内容页：草稿箱列表

约束：

- 聊天前视图是“进入聊天的入口层”，不是另起一套复杂后台
- 聊天前视图应服务于进入或恢复对话，不应偏离 Agent 场景主线
- 聊天前视图在浏览器缩放后仍需保证主入口模块完整可达；如果高度不足，应在该视图自己的内容区滚动，而不是把模块裁出页面

## 聊天头部规范

聊天头部必须承担统一职责：

- 告知用户当前处于哪个场景
- 告知当前助手身份或当前阶段
- 提供明确返回动作

统一要求：

- 头部保留 avatar / 标识区
- 头部保留标题与一行解释
- 头部保留单一返回按钮
- 返回按钮语义必须清晰，如“返回”“返回表单”

不建议：

- 在头部同时塞入过多次级按钮
- 把保存、刷新、删除等消息级动作放进头部

这些动作应属于消息级或列表级，不属于聊天头部。

## 消息流规范

### 1. 用户消息

用户消息统一规则：

- 放在右侧
- 使用气泡样式
- 宽度受限，不铺满整行
- 内容渲染统一走 [MarkdownText.tsx](../../../../frontend/components/MarkdownText.tsx)

这意味着：

- 即使用户输入本质上是纯文本，也按统一轻量 markdown 渲染
- 不要在某页直接裸渲染字符串、另一页再套解析器

### 2. assistant 消息

assistant 消息统一规则：

- 放在左侧
- 主体用浅色卡片承载
- 消息文本默认也走 `MarkdownText`
- 如果存在结构化结果，则由场景专属 renderer 替代纯文本主体

当前已存在两种 assistant 内容类型：

- 普通讨论态文本
- 结构化业务结果

前端必须以后端结果态为准消费，而不是自行猜测“这段像不像成品”。

当前消息区允许出现的正式内容类型一共三类：

- 用户消息
- assistant 正常内容
- Agent 状态反馈

### 2.1 Agent 状态反馈

生成中、停止输出、错误提示，在 harness 里统一归类为 `状态反馈块`。

当前正式纳入这一类的状态有：

- `running`
- `stopped`
- `error`

未来如果新增“重试中”“等待工具返回”“超时”等运行态，也应优先并入这一类，而不是重新发明新的提示协议。

语义边界：

- 它不是用户消息
- 它不是 assistant 正常正文
- 它不是结构化结果
- 它不是保存成功 / 删除成功这类页面级反馈

展示要求：

- 三条正式业务主线页面必须复用同一种状态反馈壳层组件或同等抽象
- 已进入聊天态时，状态反馈应出现在当前会话消息流内，贴近最新一轮上下文
- 尚未进入聊天态时，状态反馈应挂在当前入口区或输入区附近，不应先切入聊天态再补一条伪消息
- 三条正式业务主线页面的宽度、圆角、内边距、字号、对齐方式、基础排版必须一致
- 允许变化的只有文案、tone、图标或小型状态标识
- 状态反馈不挂 `MessageActions`
- 状态反馈不承担保存动作
- 状态反馈不应伪装成普通 assistant 文本写入可保存消息历史
- `error` 状态在存在可重试上下文时，必须在状态组件内直接提供“刷新 / 重试上一次请求”按钮
- 这个按钮的语义不是整页刷新，而是用同一请求参数重新发起上一条失败的 Agent 请求
- 首次请求失败时，如果页面仍停留在聊天前视图，状态组件中的刷新按钮也必须留在当前视图内完成重试，不应通过整页 reload 兜底

### 3. 消息 key 与顺序

消息流必须按真实轮次顺序显示，不允许为了 UI 拼接而打乱顺序。

统一要求：

- `user -> assistant` 是一轮的最小单位
- 刷新 assistant 回复时，回退到该 assistant 之前的历史，再重新生成
- 不要在聊天流中插入和真实轮次无关的装饰性伪消息
- `Agent 状态反馈` 可以出现在消息流里，但必须使用统一状态壳，而不是伪装成普通 assistant 正文

### 4. 场景专属页面动作按钮

聊天页允许存在少量 `场景专属页面动作按钮`，用于触发当前场景里的附加动作。

它与 `Agent 状态反馈` 的边界必须明确：

- 它不是状态提示
- 它不是消息级 `MessageActions`
- 它不是结构化结果内容本身
- 它是页面级、当前场景专属的主动动作入口

当前正式允许的例子只有：

- 热门追踪聊天页输入框上方的 `总结实时进度`

展示要求：

- 允许与 `AgentStatusMessage` 复用同一套基础视觉壳层语言
- 基础配色、字号、圆角、边框、内边距、阴影应与状态壳层保持一致，避免页面内出现第二套突兀按钮体系
- 但它必须作为新的按钮类型存在，不能直接伪装成 `AgentStatusMessage`
- 它不进入消息流
- 它不写入 `conversationHistory`
- 它只在所属场景出现，不提升为三页通用协议

## 文本输出规范

### 1. 统一使用轻量 markdown 展示

普通文本输出统一通过：

- [frontend/components/MarkdownText.tsx](../../../../frontend/components/MarkdownText.tsx)

当前基础展示能力：

- 段落
- 列表
- 1 到 3 级标题
- `**粗体**`
- `` `行内代码` ``

要求：

- 后续三页的普通文本块都继续复用 `MarkdownText`
- 不要某页自己手搓一套文本样式解析
- 不要把结构化结果强行塞回长纯文本里展示
- Markdown 语法支持范围、AI 输出归一化范围和后续增强顺序，统一以 [markdown-rendering-guidelines.md](../../../../docs/harness/03-implementation/frontend/markdown-rendering-guidelines.md) 为准

### 2. 文本内容应服务于“讨论态”而不是冒充结构化结果

普通 assistant 文本用于：

- 解释
- 追问
- 讨论
- 澄清
- 引导下一步

不用于：

- 偷偷承担结构化结果主展示
- 用一大段自然语言替代本该有 schema 的结果块

如果后端已返回结构化结果，前端应优先展示结构化块。

## 结构化结果规范

### 1. 人设页

当前人设页仍处于“文本承载结构信息”的过渡形态。

现状：

- `persona.analyze` 和 `persona.follow_up` 已进入正式后端链路
- 前端仍通过 `formatPersonaAnalyzeReply()` / `formatPersonaFollowUpReply()` 将结果整理成 markdown 文本
- 尚未像热门页、内容页那样落成独立结构化渲染块

当前要求：

- 人设页仍按当前文本方案实现
- 但必须维持统一消息壳与动作区
- 后续如果正式引入 `personaDraft` renderer，应优先替换文本拼装，而不是再叠一层历史兼容 UI

#### 保存人设重入

人设页初始态可以展示已保存人设的概要图。该概要图只用于快速扫读，不是完整人设正文。

概要图和完整保存数据的边界：

- 概要图展示的是简略人设，例如人设定位、基础画像、关键词、目标受众、内容语气、保存时间。
- 本地实际保存的是完整结构化人设数据，例如 `persona`、`niche`、`audience`、`contentStyle`、`referenceCreatorDirections`、`basicInfo`、`keywords`、`personaPosition`、`contentTone`。
- 前端不得用概要图内容反推完整人设。
- 前端不得为了重入展示调用 Agent 重新编写或补齐人设。

概要图右下角允许提供 `继续完善` 按钮。点击后的实现必须符合：

1. 只读取本地已保存的完整结构化人设。
2. 只切换 `viewMode` 到聊天态。
3. 只在消息流中放入一条 assistant 消息，内容为“上一次保存的结构化人设草稿”。
4. 不调用 `persona.analyze`。
5. 不调用 `persona.follow_up`。
6. 不展示保存对象里的历史 `conversation`。
7. 不展示 `nextQuestions`、`followUpQuestions` 或“后续可继续补充问题”。
8. 缺字段时直接不展示，不由前端生成新字段。

用户进入该聊天态后，如果继续输入修改意见，才按正常人设追问流程调用 `persona.follow_up`。

### 2. 热门追踪页

热门页存在 `completeAnalysis` 时，assistant 卡片必须渲染结构化分析块，而不是只展示一段长文本。

当前真实结构包括：

- 近一周热点趋势
- 受众需求洞察
- 可执行选题
- 初始页卡片预览 `cardPreview`

规范要求：

- 结构化分析块应保留分节标题
- `topics` 应以列表式内容展示
- 与结构化分析对应的 assistant 消息允许保存
- 聊天前历史卡片优先展示 `cardPreview.discoveryKeywords` 和 `cardPreview.shortTopics`
- 历史卡片不应展示长段 `trends`、`audience` 或自然语言摘要
- 前端不得把长摘要直接截断成卡片内容；旧数据缺少 `cardPreview` 时，只允许做 2-3 个短语级兜底

### 3. 内容撰写页

内容页存在 `completeDraft` 时，assistant 卡片必须渲染草稿结果块。

当前真实结构包括：

- 笔记标题
- 引入
- 正文内容
- 结尾互动
- 标签建议
- 初始页卡片预览 `cardPreview`

对应 renderer：

- `DraftRenderer`

规范要求：

- 不要退化回单纯把整篇笔记拼成一大段文本
- 结构化字段展示顺序应稳定
- 标签区应保持独立视觉块，而不是混回正文段落
- 草稿箱卡片主标题应展示保存后的笔记标题，不应继续显示用户最初主动输入的主题
- 草稿箱卡片优先展示 `cardPreview.keywords`
- 草稿箱卡片不应展示长段 `intro`、`hook` 或 `body`
- 前端不得把正文直接截断成卡片摘要；旧数据缺少 `cardPreview` 时，只允许从 `title`、`tags` 或既有短字段做 2-3 个短语级兜底

## 初始页卡片预览规范

人设打造、热门追踪和内容撰写的聊天前视图都有概要图 / 历史卡片 / 草稿卡片。这些卡片只承担快速扫读职责，不承担完整结果展示职责。

统一规则：

- 新生成的结构化结果必须自带 `cardPreview`。
- `cardPreview` 由 Agent 在完整结果同次输出。
- 后端负责保存和恢复 `cardPreview`。
- 前端只消费 `cardPreview`，不负责从长文本中重新总结。
- 缺少 `cardPreview` 的旧数据可以兜底，但兜底只能产生 2-3 个短语。

人设打造概要图展示：

```text
人设定位
成长型学习博主

基础画像
女 · 21 · 大学生

关键词
学习规划 · 考证经验 · 效率
```

人设打造补充规则：

- 概要图可以来自 `PersonaJson.cardPreview` 或完整保存数据中的短字段。
- 概要图不等于完整人设。
- 从概要图的 `继续完善` 进入聊天态时，必须展示完整保存人设中的结构化草稿，而不是展示概要图短字段。
- 该重入动作不触发 Agent，不生成新内容，不展示历史 conversation，不展示后续补充问题。

热门追踪卡片展示：

```text
本次发现
平价替代、新手化妆、通勤妆

适合选题
新手通勤妆、百元彩妆清单、底妆避坑
```

内容撰写草稿卡片展示：

```text
内容关键词
考证避坑、自律补救、低成本学习
```

## 消息动作区规范

消息下方统一复用：

- [frontend/components/MessageActions.tsx](../../../../frontend/components/MessageActions.tsx)

职责只有两个：

- `onRefresh`
- `onSave`

规则：

- 刷新是消息级动作，只作用于当前 assistant 回复
- 保存是结构化结果级动作，不应抽象成整个页面的全局按钮
- 如果某条 assistant 消息不具备保存意义，可以不传 `onSave`

当前场景差异：

- 人设页：assistant 消息都可保存为当前人设会话快照
- 热门页：只有携带 `analysis` 的 assistant 消息可保存
- 内容页：只有最后一条携带 `draft` 的 assistant 消息可保存

允许存在这种差异，但动作入口组件必须统一。

## 建议项规范

当前只有内容页使用正式建议项 chips。

规则：

- 建议项只出现在存在结构化草稿的 assistant 消息之后
- 建议项来自后端 / Agent 返回，不由前端硬编码业务规则拼装
- 点击建议项后，本质上等价于发起一次 revise 指令

这意味着：

- suggestions 是正式返回结构
- 不是页面临时做的快捷按钮

如果未来热门页、人设页也要扩展“下一步建议”，也应使用同一语义，不要重新发明一套按钮协议。

## 输入区规范

### 1. 输入区必须固定在聊天页底部

输入区是多轮对话主入口，必须稳定出现在聊天页底部。

统一要求：

- 使用单行输入框为主
- 右侧有单一发送按钮
- placeholder 文案体现当前场景输入预期

当前差异：

- 人设页：输入对当前人设的意见
- 热门页：输入想继续追踪的方向
- 内容页：输入主题或修改意见

允许差异只体现在 placeholder 和提交语义，不体现在整体结构。

### 1.1 生成中输入锁定属于通用设置

当 Agent 正在生成当前轮回复时，输入区必须遵守统一规则：

- 输入框保持可编辑，不应禁用
- 允许继续输入新内容，但新内容只保留在输入框中，不应直接发出
- 不允许继续按 Enter 触发发送，也不应把停止动作绑定到 Enter
- 不允许同一时刻并发发起第二个生成请求

这条规则属于三个聊天页的通用设置，不允许某页锁定、某页仍可继续编辑。

### 1.2 发送按钮在生成中必须切换为停止语义

当 Agent 处于生成态时，发送按钮不再表示“发送”，而表示“停止当前生成”。

统一要求：

- 空闲态显示发送语义
- 生成态切换为停止语义
- 点击后中断当前请求
- 中断后恢复输入能力，再允许下一轮输入

如果产品当前使用图标切换，如 `➤ -> ◯`，则这一交互语义应视为三条正式业务主线共享规则，而不是单页特例。

### 1.3 停止生成后的状态恢复

用户手动停止生成后：

- 当前请求应被取消
- 页面应退出生成态
- 输入框应重新可编辑
- 发送按钮应恢复为发送态
- 应保留一条 `stopped` 状态反馈，如“本次输出已停止”
- 不应把这条提示伪装成普通 assistant 文本消息

该行为同样属于通用设置，三页必须一致。

### 1.4 当前会话消息上限

正式聊天页必须把 `80` 条消息作为当前会话硬上限，而不是无限对话后裁剪旧消息继续发送。

统一规则：

- 消息数按 `user + assistant` 正式聊天消息合计计算。
- `Agent 状态反馈` 不作为正式聊天消息计数。
- 当当前会话消息数 `>= 79` 时，前端必须禁止继续发送新用户消息，因为需要为“用户消息 + Agent 回复”预留完整一轮空间。
- 到达上限后，输入框必须禁用，发送按钮必须禁用。
- 到达上限后，输入框 placeholder 必须显示统一文案：`当前对话已达到 80 条消息上限，请先保存或开启新对话后继续。`
- 到达上限时，页面顶部必须使用现有顶部提醒 / 报错同类样式显示一次统一文案：`当前对话已达到 80 条消息上限，请先保存或开启新对话后继续。`
- 顶部提醒只在当前会话首次达到上限时显示一次；开启新对话、返回入口并清空当前会话、或其他明确重置当前会话的动作必须重置该提醒状态。
- 该上限不新增输入区提示条，也不新增和场景按钮重复的 CTA。
- 热门追踪页点击 `总结实时进度` 完成后，如果当前会话仍达到上限，也不得恢复继续对话能力。

该规则属于正式聊天页通用设置，修改时必须同步检查 `profile`、`trending`、`content` 和 `/?view=dialog`。

### 2. 发送逻辑必须和当前 `viewMode` 对齐

规范要求：

- 非聊天态时，提交动作负责“创建一轮新会话”
- 聊天态时，提交动作负责“继续当前会话”

当前映射：

- 热门页：`history -> chat`
- 内容页：`drafts -> createDraft`，`chat -> reviseDraft`
- 人设页：`form -> analyze`，`chat -> follow_up`

## 加载态规范

加载态是 `Agent 状态反馈` 的一种，必须是当前会话上下文中的轻量状态，而不是全屏阻断。

统一要求：

- 使用与 `stopped`、`error` 相同的状态反馈壳层
- 已进入聊天态时，在聊天流下方或附近展示“Agent 正在处理中”的轻量提示
- 尚未进入聊天态时，在当前入口区或输入区附近展示同类状态提示
- 不用 modal，不抢占整个页面
- 加载态文案允许按场景写具体，但语气应一致
- 加载提示不进入普通 assistant 消息历史

当前现状：

- 三条正式业务主线页面都已有轻量加载提示
- 文案尚未完全统一

这轮先记录为现状，不强行改成同一句。

补充要求：

- 加载提示可以按场景写不同文案
- 但加载时“禁止继续发送、允许停止当前生成”的交互语义必须统一

## 错误态规范

错误态也是 `Agent 状态反馈` 的一种。当前三条正式业务主线页面对错误态的实现并不完全一致，但应遵守同一原则：

- 错误必须反馈给用户
- 错误优先保留在当前会话上下文里
- 不要因为请求失败就悄悄吞掉本轮输入

以下规则属于通用设置：

- 对话生成相关错误应优先使用统一 `Agent 状态反馈` 壳，不再直接混成普通 assistant 文本
- 如需额外页面级提醒或动作按钮，可补充统一顶部提示组件，但它只能作为补充，不替代主状态反馈
- 顶部错误提示如果提供动作按钮，其语义和位置在三条正式业务主线页面必须一致
- 当前统一动作可为“刷新”
- 错误提示不应混入成功提示样式，也不应退化为 `window.alert`

当前已存在的实现方式：

- 向消息流追加 `出错了：...`
- 顶部或局部 notice
- `window.alert`

规范方向：

- 长期应逐步收敛到统一 `Agent 状态反馈`
- 但在统一前，至少要保证错误可见、可追溯、不会把会话直接清空

新增硬规则：

1. 网络错误或接口错误不能悄悄切回默认示例界面、mock 视图或伪数据视图。
2. 如果失败发生在“进入聊天前的首次请求”：
   - 应保留在当前入口页
   - 使用同一类 `Agent 状态反馈` 直接展示错误
   - 不应先切入聊天态再显示失败
3. 如果失败发生在已进入聊天态后的继续对话：
   - 应保留当前会话上下文
   - 不应把会话重置成入口页
   - 不应把错误伪装成可保存的 assistant 正文
4. 错误态处理属于通用设置，修改时必须同步检查三条正式业务主线页面。
5. 错误态的“刷新”按钮必须绑定为“重试上一次请求”，不能绑定为 `window.location.reload()` 这类整页刷新动作。

## 成功反馈规范

保存成功 / 删除成功必须复用：

- [frontend/components/TopToast.tsx](../../../../frontend/components/TopToast.tsx)

具体规范见：

- [frontend-feedback-guidelines.md](../../../../docs/harness/03-implementation/frontend/frontend-feedback-guidelines.md)

本文档只补充页面层要求：

- 顶部成功提示不进入聊天消息流
- 成功提示不替代真实保存状态更新
- 成功提示只用于轻量完成反馈
- 成功提示不属于 `Agent 状态反馈`
- 成功反馈与错误反馈都属于页面级通用设置，不能只改其中一页

## 会话现场恢复规范

聊天页属于 session 级现场，不是正式业务结果本身。

统一要求：

- `viewMode` 要可恢复
- 消息流要可恢复
- 滚动位置要可恢复
- 当前激活对象要可恢复
- 正在 loading 的旧状态恢复时应谨慎处理

当前真实实现：

- 人设页恢复 `info / messages / draftMessage / scrollTop`
- 热门页恢复 `analysis / messages / conversationHistory / scrollTop`
- 内容页恢复 `activeDraft / activeTopic / conversationHistory / scrollTop`
- 灵光一闪恢复 `chatHistory / agentStatus / scrollTop`

实现约束：

- session 恢复逻辑应属于页面壳层，不应混进 renderer
- 恢复的是“会话现场”，不是“强行认定业务已保存”

### 灵光一闪本地会话恢复

灵光一闪是轻量通用聊天页，不进入后端业务保存链路，但它仍属于正式聊天页面体系，必须保留当前浏览器里的单条会话现场。

当前实现要求：

- 使用浏览器 `localStorage` 保存当前单条灵光一闪会话，而不是只保存在组件内存中。
- 保存内容包括消息流、Agent 状态反馈和滚动位置。
- 从 `/?view=dialog` 进入时，如果存在本地会话，应恢复到聊天态并继续展示历史消息。
- 发送 `/api/chat` 时，恢复后的消息流仍应作为 `conversationHistory` 参与上下文。
- 同一前端运行期间，如果 Agent 正在输出时用户切换到其他页面，请求必须继续运行；完成、失败或停止后必须直接更新本地会话现场。
- 用户回到 `/?view=dialog` 时，应看到当前请求仍在 loading，或看到后台完成后的最新 assistant 回复。
- 只有刷新页面、重开浏览器或运行时上下文丢失导致原请求无法继续时，前端才应退出 loading，并展示可见的中断状态反馈，避免用户卡在等待态。
- 该本地恢复不等同于“保存记录”，不产生历史列表、不跨设备、不写入账号数据。

灵光一闪聊天态头部允许出现 `新建` 页面动作：

- `新建` 的位置沿用聊天头部右侧动作区，视觉层级参考热门追踪聊天页的返回按钮。
- 未登录时，`新建` 与登录入口并排显示，不互相替代。
- 点击 `新建` 必须中断当前请求、清空输入框、清空消息流、清空状态反馈、清空滚动位置，并删除本地会话存储。
- 点击后页面应保持或跳转到 `/?view=dialog`，展示灵光一闪初始输入页。
- `新建` 是页面级重置动作，不进入消息流，不写入 `conversationHistory`，也不代表后端新建业务记录。

## 路径迁移规范

迁移文档、测试夹具、静态资源或配置文件时，默认视为“可能影响使用方”的动作，不能只移动文件不检查引用。

统一要求：

- 迁移前先搜索旧路径是否被代码、测试、文档或脚本引用
- 迁移后同步更新引用方路径
- 迁移完成后再复查一次，确认没有断链、失效引用或读取失败

高风险迁移类型包括：

- 测试 JSON / fixtures
- markdown 文档之间的互相引用
- 页面依赖的图片、静态资源、配置文件
- 代码中写死路径的读取逻辑

结论：

- 路径迁移不是纯整理动作
- 只要存在引用，就必须把“是否影响路径”视为实现改动的一部分

## 前置依赖门禁规范

不是所有聊天页都可无条件进入。

当前门禁规则：

- 热门页和内容页必须先有人设
- 统一通过 [frontend/components/RequirePersona.tsx](../../../../frontend/components/RequirePersona.tsx) 承载

规范要求：

- 需要前置条件的聊天页，优先通过统一 guard 组件处理
- 不要在页面内部零散写多段 if/else 占据主渲染逻辑

## 允许差异与禁止差异

### 允许差异

- 聊天前视图类型可以不同
- 结构化结果 renderer 可以不同
- 输入 placeholder 和文案可以不同
- 保存目标和保存条件可以不同
- 是否展示 suggestions 可以不同

### 禁止差异

- 不允许每页发明不同的消息主骨架
- 不允许每页发明不同的 assistant 基础卡片协议
- 不允许每页各自解析一套文本渲染器
- 不允许把消息级动作随意搬到头部或列表区
- 不允许前端绕过后端结果态自行判定业务完成语义

## 对前端实现者的直接要求

后续新增或修改聊天页时，默认遵守以下规则：

1. 先复用现有聊天页骨架，再做场景差异化。
2. 先定义消息态、结构化结果态、动作态，再写样式。
3. 普通文本统一走 `MarkdownText`，动作统一走 `MessageActions`，轻量成功反馈统一走 `TopToast`。
4. 会话现场恢复视为页面基础能力，不视为可选增强。
5. 前端只消费后端结果态，不负责重新解释业务是否完成。
6. 任何“生成中交互”“停止生成”“错误提示”“成功提示”修改，都先按通用设置处理，不得只改单页。

## 当前文档落地结论

对当前三条正式业务主线页面，可以把前端 harness 规范总结为一句话：

```text
统一页面骨架，统一消息语义，统一动作入口，结构化结果按场景分化，页面状态按 session 恢复。
```

后续如果要继续推进统一，优先顺序应为：

1. 人设页补齐正式结构化结果 renderer
2. 三条正式业务主线页面的错误态与加载态进一步收敛
3. 把当前页面骨架再抽成更明确的共享聊天壳组件
