# Harness 测评与异常归档方案

## 目标

本文档定义当前 harness 体系下的测评方案，覆盖：

- 全量测评
- 跨模块回归测评
- 模块测评
- 异常测评
- 错误记录与检索格式

本文档的目的不是只列一个 checklist，而是形成一套可重复执行、可复盘、可检索相似问题的测评体系。

## 适用范围

当前一期重点覆盖：

- 前后端连接
- Agent 调用
- 网络 Agent 调用
  - 当前默认未启用
  - 启用后纳入全量测评
- 登录与 session 用户隔离
- 人设打造
- 热门追踪
- 内容撰写
- 灵光一闪 `/?view=dialog`

说明：

- 人设、热门、内容是当前三条正式业务主线。
- 灵光一闪属于正式聊天页面体系，但当前仍按轻量通用聊天页验收。
- `/data`、`/operation` 不在当前正式聊天主线测评范围内。

## 测评分层

当前测评分为四层：

1. 全量测评
2. 跨模块回归测评
3. 模块测评
4. 异常测评

统一要求：

- 每项测评都应明确触发动作
- 每项测评都应明确预期结果
- 每项异常都应形成结构化记录

## 测试工具与位置

为避免“知道要测，但不知道工具在哪”的情况，当前 harness 默认按下面理解：

- 后端接口与服务层测试
  - 工具：`pytest`
  - 代码位置：`backend/tests/`
  - 依赖位置：`backend/requirements-test.txt`
  - 推荐环境：Docker `backend` 容器
- Agent workflow / runtime / tool 测试
  - 工具：`pytest`
  - 代码位置：`agent/tests/`
  - 依赖位置：`agent/pyproject.toml`
  - 推荐环境：Docker `agent` 容器
- 前端真实浏览器实操测试
  - 工具：`playwright`
  - 浏览器：`chromium`
  - 当前脚本位置：`frontend/scripts/ui-fulltest.mjs`
  - 依赖位置：`frontend/package.json`

补充说明：

- 本文档中如果提到“浏览器实操测试”，默认不是随手打开浏览器看一下，而是指可重复执行的页面级真实交互验证。
- 当前仓库里前端这一类测试工具口径是 `Playwright + Chromium`；如果后续增加别的浏览器或 E2E 框架，需要在本文档同步补充。
- 当前本机未必直接安装了所有测试依赖；若本地缺失，优先去对应 Docker 容器内执行，而不是默认切回宿主机裸跑。

## 固定测试名

### 输出内容测试

`输出内容测试` 作为当前 harness 体系中的固定测试名，特指以下这一类检查：

- 检查 mock / model 阶段输出是否正常
- 检查 discussion 与非 discussion 输出是否正常
- 检查后端对 Agent 输出的包装是否仍符合 harness 统一结果态语义

这项测试的目标不是只看“有没有返回”，而是确认返回内容是否仍然落在当前 harness 约定的两类主结果态中：

- `discussionOnly = true`
  - 只返回讨论态文本
  - 不应误出现保存入口
  - 不应误进入结构化结果保存链路
- `discussionOnly = false`
  - 应返回完整结构化结果
  - 热门追踪应表现为 `completeAnalysis` 一类结果
  - 内容撰写应表现为 `completeDraft` 一类结果

同时还要检查主结果与伴随字段是否一致，例如：

- `text` 是否与当前结果态一致
- `raw` 是否保留了可追查的原始输出
- `suggestions` 是否只在应出现的场景和结果态中出现
- 前端最终渲染是否与后端包装后的语义一致

统一命名约定：

- 后续在 harness 文档中，优先统一使用 `输出内容测试`
- 旧文里的“结果态类”“结果测评”“结果态判定”等说法，都默认视为 `输出内容测试` 的组成部分
- “内容测试”如果专指输出是否为讨论态或结构化结果，也归并到 `输出内容测试`，不再单独作为并列测试名使用

## 1. 全量测评

全量测评用于确认整条主链路没有断，适合：

- 大版本改动后
- 改动后端 Agent 调用链路后
- 改动多模块共享逻辑后
- 提测前

执行口径（重要）：

- 本文档中的全量测评，默认包含真实浏览器实操测试。
- 这里的“真实浏览器实操测试”指由执行者（包括 AI Agent）代替人工完成页面打开、点击、输入、提交、等待返回、结果校验等完整交互动作。
- 只跑 `pytest`、`npm test`、接口直调，不可视为完成本文档定义的全量测评；这些仅属于全量测评的一部分支撑项。
- 正式页面范围默认包含：`/profile`、`/trending`、`/content`、`/?view=dialog`。

### 1.1 系统链路类

需要覆盖：

- 前端可以正常请求后端
- 后端可以正常调用 Agent
- Agent 返回结果可被后端正确整理
- 网络 Agent 调用能力启用时可正常调用
- 网络 Agent 不可用时可正确降级或提示
- mock / model 模式能按当前配置跑通

建议测项：

- `POST /api/auth/register` 可调用，并在成功后建立有效登录态
- `POST /api/auth/login` 可调用
- `GET /api/auth/me` 可正确返回 401 或当前用户
- `POST /api/auth/logout` 可调用，并使当前前端状态回到未登录态
- 首页进入后基础接口无报错
- `POST /api/chat` 可调用
- `POST /api/persona/analyze` 可调用
- `POST /api/trends/track` 可调用
- `POST /api/content/draft` 可调用
- Agent 不可用时能返回清晰错误
- 网络搜索启用后，热门追踪能获得带来源或带 warnings 的结果

认证闭环补充要求：

- 注册成功后应提示用户继续登录，不要求前端直接进入已登录态
- 带 `next` 的登录 / 注册完成后，应回到原目标页
- 退出登录后再次调用 `/api/auth/me` 应返回未登录态

### 1.2 场景交互类

每个模块至少覆盖一次正常主流程：

#### 人设打造

- 未登录状态可完成 1 次人设生成
- 未登录状态第 2 次人设生成被前端拦截并提示登录
- 未登录状态不支持保存人设
- 首轮发起人设分析
- 继续追问
- 保存人设
- 保存后可正确读取当前账号已保存人设
- 浏览器缩放到 `125%`、`150%` 时，人设打造初始模块不应被直接裁出页面
- 浏览器缩放后若“查看已保存人设信息”展开内容超出当前可视高度，应在当前场景内容区出现可用滚动，而不是让底部内容不可达

#### 灵光一闪

- 首页进入 `/?view=dialog` 后可正常发起 `general.chat`
- 能展示普通 assistant 回复
- 能继续多轮对话
- 切换到其他页面后再回到 `/?view=dialog`，仍显示本地聊天历史
- Agent 输出中切换到其他页面，请求应继续运行，回到 `/?view=dialog` 后能看到仍在加载或已完成的最新回复
- 刷新页面后仍显示本地聊天历史
- 重开浏览器后仍显示本地聊天历史
- 点击聊天态头部的“新建”后，清空本地聊天历史并回到灵光一闪初始页
- 点击“新建”后再次刷新，不应恢复旧聊天历史
- 不应出现结构化结果 renderer
- 不应出现保存入口
- 不应误出现热门 / 内容 / 人设业务结果卡片
- 未登录状态下仍可正常使用轻量通用聊天能力

#### 热门追踪

- 未登录进入热门追踪时显示登录引导，不发起业务请求
- 已登录但无人设时显示“先完成人设打造”引导
- 创建新的热门追踪对话
- 正常得到讨论态回复
- 首条业务输入直接得到完整分析态回复
- 正常得到后续讨论态回复
- 保存追踪结果
- 删除追踪记录
- 继续旧追踪记录对话
- 在旧追踪记录上再次保存
- 触发 `summaryMode = realtime_progress` 后返回新的完整 `completeAnalysis + cardPreview`
- 再次保存后历史卡片与当前结构化结果同步更新
- 保存成功后页面顶部中间出现成功提示，约 2 秒后自动消失
- 删除成功后页面顶部中间出现成功提示，约 2 秒后自动消失

#### 内容撰写

- 未登录进入内容撰写时显示登录引导，不发起业务请求
- 已登录但无人设时显示“先完成人设打造”引导
- 创建新的内容对话
- 只讨论方向，不输出笔记
- 首条业务输入直接输出完整笔记
- 保存草稿
- 删除草稿
- 打开旧草稿继续对话
- 在旧草稿上再次保存
- 点击 suggestions 继续 revise
- 带 `currentDraft` 的请求实际进入 `content.revise`
- revise 后再次保存时更新原草稿，而不是新增错乱记录
- 保存后草稿箱标题与 `cardPreview.keywords` 同步更新
- 保存成功后页面顶部中间出现成功提示，约 2 秒后自动消失
- 删除成功后页面顶部中间出现成功提示，约 2 秒后自动消失

### 1.3 会话与状态类

需要覆盖：

- 注册成功后提示继续登录，并在登录完成后回跳目标页
- 登录后拉取当前用户的人设、趋势历史、草稿历史
- 退出登录后 `/api/auth/me` 重新变为未登录态
- 退出登录后清空当前前端用户数据
- 切换账号后不串用上一账号的数据
- 新建聊天记录
- 删除聊天记录
- 继续旧聊天记录
- 刷新页面后恢复当前会话
- 切换视图后状态保持正确
- 灵光一闪本地单会话恢复不应和后端业务保存混淆
- 灵光一闪输出中切换页面不应中断后台请求，也不应在回到页面时丢失最终回复
- 灵光一闪新建重置后不应残留旧 `localStorage` 聊天现场
- 未保存结果与已保存结果不混淆
- 删除当前活跃记录后页面正确回退

### 1.4 输出内容测试

这是当前 harness 主线下必须重点测的部分，也就是前文定义的 `输出内容测试` 在全量测评中的主体。

需要覆盖：

- `discussionOnly` 是否被正确识别
- 结构化结果态是否被正确识别
- 结构化结果态是否可以保存
- 讨论态是否不会误进入保存链路
- 前端是否按结构化结果正确渲染
- 热门追踪与内容撰写首条业务输入是否直接进入完整结构化路径
- 通用聊天 `general.chat` 是否始终保持普通文本结果，不误进入结构化业务结果路径

场景要求：

#### 热门追踪

- 讨论态不出现结构化分析保存入口
- 完整分析态出现结构化分析与保存入口
- 首条业务输入不应退回 discussionOnly
- 完整分析最低标准符合：
  - `trackName`
  - `trends`
  - `audience`
  - `topics >= 1`
- `cardPreview.discoveryKeywords` 与 `cardPreview.shortTopics` 存在且可用于历史卡片

#### 内容撰写

- 讨论态只显示对话文本
- 完整草稿态显示 `completeDraft`
- 完整草稿态显示 suggestions
- 点击 suggestions 能继续 revise
- 首条业务输入不应退回 discussionOnly
- `cardPreview.keywords` 存在且可用于草稿卡片
- `intro` / `hook` 兼容读取时不导致渲染失败

#### 灵光一闪

- `general.chat` 只返回普通文本结果
- 不出现保存入口
- 不误进入结构化结果渲染链路

## 2. 跨模块回归测评

这层专门用于解决“在 A 页修好，B 页又复发”的问题。

适用于：

- 修改登录态、门禁或退出逻辑
- 修改共享交互模式
- 修改统一结果态语义
- 修改保存 / 删除 / 恢复逻辑
- 修改聊天页共性组件

需要统一核对：

- 注册成功后的提示登录与跳转是否一致
- 注册成功后的提示登录与跳转是否一致
- 登录按钮位置是否一致
- 热门追踪 / 内容撰写门禁是否一致
- 退出登录入口和行为是否一致
- 灵光一闪是否仍保持“可聊天但不保存、不结构化”的统一边界
- 输入框行为是否一致
- 发送后 loading 是否一致
- 生成中是否允许继续编辑输入框且不直接发出
- 生成中是否只能通过点击圆圈停止当前输出
- `输出内容测试` 相关语义是否一致
- `discussionOnly` 与结构化结果逻辑是否一致
- 浏览器缩放后核心模块是否仍完整可达，必要时是否只在本场景内容区滚动而不是整页裁切
- 保存动作是否一致
- 删除动作是否一致
- 保存成功反馈是否一致
- 删除成功反馈是否一致
- 继续旧记录的恢复逻辑是否一致
- 错误提示风格是否一致

成功反馈统一规范：

- 保存成功或删除成功后，应在页面顶部中间显示轻量提示，不占用主内容区排版
- 提示采用短时浮层形式，带轻微侧滑入场与退场效果
- 提示默认展示约 2 秒后自动消失
- 提示文案应简洁直接，优先使用“保存成功”“删除成功”
- 同类成功提示在 `persona`、`trend`、`content` 页面中的位置、时长和动画应保持一致

## 3. 模块测评

模块测评用于针对当前正在修改的模块或功能做定向验证。

每次模块测评建议按六类执行：

1. 认证 / 门禁测评
2. 页面交互测评
3. 状态恢复测评
4. 输出内容测试
5. 保存 / 删除 / 恢复测评
6. UI 测评

### 3.1 认证 / 门禁测评

检查：

- 登录按钮是否跳到 `/login?next=当前页`
- 注册成功后是否提示继续登录
- 登录成功后是否返回原目标页
- 退出登录后是否回到未登录态
- 游客第 2 次人设生成是否被拦截
- 未登录时受保护页面是否只显示门禁，不误发业务请求
- session 过期后是否正确回到未登录态

### 3.2 页面交互测评

检查：

- 按钮是否可点
- 输入后是否能触发请求
- 生成中输入框是否仍可编辑
- 生成中新输入是否不会直接发出
- 生成中点击圆圈是否只停止当前输出
- 停止后是否可以继续发送刚刚输入的新内容
- 返回上一级视图是否正常
- 灵光一闪是否能稳定进行多轮 `general.chat`
- 热门追踪 `summaryMode = realtime_progress` 是否只在所属场景动作入口触发

### 3.3 状态恢复测评

检查：

- `/api/auth/me` 变化后前端状态是否正确切换
- 未登录时是否不会恢复其他账号的远端业务数据
- 登录后是否恢复当前账号远端数据
- sessionStorage 是否正确恢复
- localStorage 是否正确恢复
- active 记录是否正确
- 删除后状态是否同步清理
- 新旧记录切换是否串数据

### 3.4 输出内容测试

检查：

- 讨论态 / 成品态识别是否正确
- 结构化结果字段是否完整
- 返回 shape 是否符合协议
- 是否误进入保存链路
- 热门 / 内容首条业务输入是否直接命中完整结构化路径
- `general.chat` 是否始终保持普通文本结果
- `content.revise` 是否在带 `currentDraft` 时走 revise 语义

### 3.5 保存 / 删除 / 恢复测评

检查：

- 保存 / 删除 / 继续对话动作是否正常
- 首次保存是否正常
- 在旧记录上再次保存是否是更新而不是脏新增
- 删除后刷新是否不复活
- 保存 / 删除成功后是否出现统一成功提示
- 历史卡片 / 草稿卡片是否同步更新
- `draftSource`、`writingEntrySource`、`cardPreview` 是否按当前协议保留

### 3.6 UI 测评

检查：

- 布局是否错位
- 长文本是否溢出
- markdown 是否正常渲染
- suggestions / topics / tags 多项时是否挤坏布局
- loading、empty、error 状态是否自然
- 移动端是否可用
- 顶部成功提示是否位于页面顶部中间，且不会遮挡关键操作区
- 顶部成功提示的出现、停留、消失是否自然

## 4. 异常测评

异常测评不只判断“有没有报错”，而要判断：

- 报错是否清晰
- 报错是否具体指出问题原因
- 是否能快速判断是哪一层的问题
- 是否有明确排查方向或解决办法

### 4.1 可预期异常

需要测：

- 未登录访问受保护业务接口
- 注册成功后未正确建立登录态
- session 过期
- Agent 服务未启动
- Agent 地址配置错误
- 后端接口超时
- 网络 Agent / web search 未配置
- 必填参数缺失
- 保存接口失败

预期：

- 错误提示可见
- 提示内容可区分问题层级
- 不出现静默失败

补充要求：

- 未登录访问受保护接口时返回统一 401 结构
- 前端门禁页不应误发起受保护业务请求

### 4.2 结构异常

需要测：

- Agent 返回字段缺失
- `completeAnalysis` 不完整
- `completeDraft` 不完整
- suggestions 结构不合法
- `intro` / `hook` 混乱导致渲染异常

预期：

- 后端或前端有兜底
- 错误能进入统一归档格式

### 4.3 状态异常

需要测：

- 退出登录后仍看到旧账号数据
- 切换账号后趋势历史或草稿历史串数据
- 删除后记录再次出现
- session 恢复覆盖新结果
- activeDraft / active record 错乱
- 已保存结果与临时会话混淆

预期：

- 可以清楚归因到 state / storage 层
- 可以稳定复现和记录

## 5. 错误记录与检索格式

错误记录不只用于当次修 bug，还用于后续检索相似问题。

因此每条错误记录都应包含：

- 问题现象
- 初步归因
- 原始报错
- 归一化错误名
- 解决办法或排查方向
- 是否已有相似历史问题

### 5.1 推荐字段

- `error_id`
- `time`
- `scenario`
- `action`
- `layer`
- `symptom`
- `raw_error`
- `normalized_error`
- `possible_cause`
- `suggested_fix`
- `resolution`
- `similar_cases`
- `repro_steps`
- `status`

### 5.2 字段说明

- `error_id`
  - 错误编号，例如 `ERR-TREND-001`
- `time`
  - 发现时间
- `scenario`
  - 所属场景，例如 `persona`、`trend`、`content`、`global`
- `action`
  - 触发动作，例如“创建新的热门追踪”
- `layer`
  - 初步归因层级
- `symptom`
  - 用户看到的现象
- `raw_error`
  - 原始报错内容
- `normalized_error`
  - 归一化错误名称
- `possible_cause`
  - 初步原因判断
- `suggested_fix`
  - 建议排查方向
- `resolution`
  - 已确认的解决办法或最终修复方式
- `similar_cases`
  - 历史相似问题编号或链接
- `repro_steps`
  - 复现步骤
- `status`
  - `new` / `confirmed` / `fixing` / `resolved` / `cannot_repro`

### 5.3 `layer` 推荐枚举

- `frontend-ui`
- `frontend-state`
- `backend-api`
- `backend-service`
- `agent-runtime`
- `agent-tool`
- `network`
- `data-storage`
- `unknown`

### 5.4 `normalized_error` 推荐枚举

- `AGENT_UNAVAILABLE`
- `AGENT_BAD_RESPONSE`
- `BACKEND_TIMEOUT`
- `BACKEND_ROUTE_ERROR`
- `NETWORK_SEARCH_UNAVAILABLE`
- `INVALID_RESULT_SHAPE`
- `SAVE_FAILED`
- `SESSION_STATE_CONFLICT`
- `DELETE_STATE_CONFLICT`
- `UNKNOWN_RUNTIME_ERROR`

### 5.5 错误记录模板

```yaml
error_id: ERR-TREND-001
time: 2026-05-18 15:20
scenario: trend
action: 创建新的热门追踪
layer: agent-runtime
symptom: 点击发送后前端提示 Agent 调用失败
raw_error: Connection refused to AGENT_BASE_URL
normalized_error: AGENT_UNAVAILABLE
possible_cause: Agent 服务未启动或端口错误
suggested_fix: 检查 agent 服务状态、端口与 AGENT_BASE_URL 配置
resolution: 启动 agent 服务，并将后端环境变量中的 AGENT_BASE_URL 改为正确端口
similar_cases:
  - ERR-GLOBAL-002
repro_steps:
  - 打开热门追踪页
  - 输入“大学生成长赛道”
  - 点击发送
status: confirmed
```

### 5.6 相似问题检索原则

后续检索相似问题时，优先按以下维度查找：

1. `normalized_error`
2. `layer`
3. `scenario`
4. `action`
5. `resolution`

这意味着错误归档的目标不是“留痕”，而是形成可复用的解决知识库。

## 6. 测试输入样例

每个场景建议至少准备：

- 正常输入
- 讨论态输入
- 成品态输入
- 边界输入
- 异常输入

### 热门追踪

- 正常输入：`我想找大学生成长赛道里更容易涨粉的选题`
- 讨论态输入：`这个方向适合我吗`
- 边界输入：`大学生`
- 异常输入：`嗯`

### 内容撰写

- 正常输入：`帮我写一篇大学生低成本自律笔记`
- 讨论态输入：`这个方向适合写吗`
- 改稿输入：`标题更像搜索词`
- 异常输入：`继续`

### 人设打造

- 正常输入：完整基础信息 + 继续追问
- 边界输入：只填年龄和职业
- 异常输入：空输入或缺关键字段

### 灵光一闪

- 正常输入：`我想做一个大学生效率提升账号，先帮我想几个可聊方向`
- 多轮输入：`第二个方向展开一点，再给我一个更生活化的表达`
- 恢复输入：完成两轮对话后切到热门追踪，再回到 `/?view=dialog`
- 后台加载输入：发起一轮较长请求后立刻切到内容撰写，等待返回后再回到 `/?view=dialog`
- 重置输入：在已有聊天历史时点击聊天态头部“新建”
- 边界输入：`你好`
- 异常输入：连续快速发送两次不同问题，并在生成中点击停止

### 认证 / 门禁

- 注册输入：新邮箱 / 用户名 + 合法密码 + `next` 目标页
- 登录输入：已存在账号 + 正确密码 + `next` 目标页
- 异常输入：错误密码、重复邮箱、缺失字段、过期 session

## 7. 执行方式标记

每条测项建议标记执行方式：

- `手测`
- `接口测`
- `自动化`
- `待自动化`

推荐：

- `输出内容测试` 优先做接口测
- UI 问题优先手测
- 记录创建 / 删除 / 恢复优先手测，后续逐步自动化

补充约定：

- 当前仓库语境下，`手测` 包含“由 AI Agent 执行的真实浏览器实操”。
- 对全量测评中的页面交互项，默认要求使用真实浏览器完成至少一轮可复现操作，不得仅以接口调用替代。

## 8. 通过标准

每项测评都应给出通过标准。

示例：

- 注册成功：出现清晰成功提示，并可继续登录后按 `next` 返回目标页
- 退出登录：前端立即回到未登录态，再调用 `/api/auth/me` 返回未登录
- 未登录热门追踪：显示登录引导，不发趋势请求
- 未登录内容撰写：显示登录引导，不发草稿请求
- 游客人设第 1 次：可生成但不可保存
- 游客人设第 2 次：前端提示登录后继续
- 未登录灵光一闪：可正常多轮聊天，但不出现保存入口、结构化卡片和业务化结果
- 灵光一闪本地恢复：切页、刷新、重开浏览器后仍保留当前单条聊天现场
- 灵光一闪后台加载：Agent 输出中切页不会中断请求，回到 `/?view=dialog` 后显示仍在加载或已完成回复
- 灵光一闪新建：清空当前本地聊天现场并回到 `/?view=dialog` 初始页，刷新后旧记录不复活
- 讨论态：只展示讨论回复，不出现保存入口
- 完整分析态：出现结构化分析与保存入口
- 热门首条业务输入：直接返回完整分析，不退回 discussionOnly
- 完整草稿态：出现完整 draft、suggestions 和保存入口
- 内容首条业务输入：直接返回完整 draft，不退回 discussionOnly
- 点击 suggestions 改稿：请求按 `content.revise` 语义更新当前草稿
- 旧记录再次保存：更新原记录及历史卡片，不新增串乱脏记录
- 删除记录：页面即时移除，刷新后不复活
- 继续旧记录再保存：结果更新符合预期，不出现状态串乱
- 保存成功 / 删除成功：顶部中间提示出现、约 2 秒后自动消失，三大业务页表现一致
- Agent 不可用：错误提示可见，且能定位问题层级
